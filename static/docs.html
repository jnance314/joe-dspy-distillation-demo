How to use the DSPy Prompt Optimizer to get production-ready prompts.

What This Tool Does

This tool helps you get high-quality results from cheap, fast AI models without manually writing prompts.

Here's the problem it solves: you have a task (like checking if marketing copy follows brand guidelines). An expensive, smart model does it well, but it's slow and costs a lot per request. A cheap model is fast and affordable, but it gets the task wrong without a carefully crafted prompt.

This tool automatically finds the best prompt for the cheap model by:

  1. Using a smart "teacher" model to generate high-quality examples of the task being done well
  2. Testing different combinations of instructions and examples to find what works best
  3. Scoring everything against your evaluation metrics
  4. Giving you the winning prompt that you can copy-paste into any API

You run the optimization once. After that, you deploy the cheap model with the optimized prompt — no teacher model needed in production.

Field Reference

Task Configuration

Task Name

A label for your task. Used for saving and loading configurations. Example: "Brand Voice Compliance - Liquid Death"

Description

A one-sentence description of what the model should do. This becomes the opening line of the system prompt that the model sees. Be specific. Example: "Check if marketing copy complies with brand voice guidelines. If non-compliant, identify problematic phrases and suggest an on-brand replacement."

Guidelines / Context

The rules, context, or reference material the model needs to do its job. This gets included in every prompt. For brand compliance, this would be your brand voice guide — tone rules, banned phrases, preferred alternatives, structural rules. The more specific you are, the better the model performs.

Signature Fields

These define the model's input/output contract — what goes in and what comes out.

Input Fields

What the model receives for each request. Each field has a name (used in the data table) and a description (tells the model what this field contains). Example: an input field called marketing_copy with description "The marketing copy to check".

Output Fields

What the model produces. Same structure as inputs — name + description. These are what get scored by your metrics. Example: compliant ("true if on-brand, false if not"), flagged_phrases ("comma-separated problematic phrases"), suggestion ("rewritten on-brand version").

Tip: Keep output field names simple and snake_case. The descriptions matter — they tell the model exactly what format you expect. "Comma-separated list" vs "JSON array" vs "one per line" will produce very different outputs.

Evaluation Metrics

Metrics define how the model's output is scored. The optimizer uses these scores to find the best prompt. Each metric targets one output field.

Metric TypeWhat It DoesBest For
Exact Match Compares the model's output to the expected value (case-insensitive). Returns 1.0 if they match, 0.0 if not. Classification fields like "true"/"false", category labels, yes/no answers.
F1 Phrases Parses comma-separated phrases from both expected and predicted, then computes precision/recall/F1 with fuzzy matching. Fields where the model lists multiple items, like flagged phrases or extracted entities.
Rule Quality Checks the output text against structural rules: no banned words, max sentence length, no passive voice. Returns a score based on how many rules pass. Free-text outputs like rewritten copy, suggestions, or generated content that must follow specific rules.
Custom Write your own Python scoring function. Must define def metric(example, pred, trace=None) -> float returning 0.0-1.0. Anything the built-in types don't cover.
Weight

How much this metric contributes to the overall score. Weights are normalized automatically (they don't need to sum to 1.0). A metric with weight 0.4 matters more than one with weight 0.3.

Target Field

Which output field this metric scores. Make sure each output field has at least one metric, or the optimizer won't know if that field is correct.

Rule Quality Options

When you select "Rule Quality" as the metric type, additional options appear:

Banned Words

One phrase per line. If any of these appear in the model's output, that check fails. Example: corporate jargon like "premium quality", "best-in-class", "leverage".

No Passive Voice

When checked, the metric flags passive voice patterns ("was created", "is being used"). Useful for brand voices that require active, direct language.

Max Sentence Length

Maximum average words per sentence. Set to 0 to disable. Useful for brands that require short, punchy copy.

Training Data

Examples Table

Labeled examples that the optimizer learns from. Each row has values for every field (inputs AND outputs). The optimizer uses these to figure out what "good" looks like.

Data Split (train / val / test)

Your examples are automatically split three ways:

  • Train — The optimizer picks the best few-shot examples from these
  • Validation — The optimizer scores candidate prompts against these during search
  • Test — Never seen during optimization. Used only for the final honest evaluation
How many examples? Aim for 30-50 minimum. More is better. Include a mix of cases — for brand compliance, that means both compliant and non-compliant copy covering different types of violations.

Model Configuration

Teacher Model

The smart, expensive model used during optimization. It generates high-quality demonstrations that teach the student model how to do the task. Only runs during optimization — never in production. Example: gemini/gemini-3.1-pro-preview

Student Models

The cheap, fast models you want to deploy in production. The optimizer finds the best prompt for each one. You can test multiple to compare. Example: gemini/gemini-2.5-flash-lite

Eval Trials

How many times to run the evaluation for each configuration. More trials = more stable numbers (mean and standard deviation). Default is 10. Use 30+ for results you want to present confidently.

Threads

Parallel API calls during evaluation. Higher = faster but may hit rate limits. Default 50 works well for Gemini.

Walkthrough: Default Task

Let's walk through the tool using the built-in Liquid Death brand voice example.

1 Load the default task

Click "Default Task" and make sure "Use default synthetic data" is checked. This pre-fills everything: guidelines, fields, metrics, and 53 labeled examples.

2 Review what's configured

Scroll through the page to see what's set up:

  • Guidelines: Liquid Death's brand voice rules — irreverent, anti-corporate, punk. Includes banned phrases ("premium quality", "leverage", etc.) and preferred alternatives ("purchase" → "grab").
  • Input: marketing_copy — the text to check.
  • Outputs: compliant (true/false), flagged_phrases (what's wrong), suggestion (the fix).
  • Metrics: Exact match on compliance (40%), F1 on flagged phrases (30%), rule quality on suggestions (30%).
  • Examples: 53 rows mixing on-brand copy and corporate jargon violations.
3 Check model configuration

The defaults work well:

  • Teacher: gemini/gemini-3.1-pro-preview ($2.00/M tokens) — the smart model
  • Student: gemini/gemini-2.5-flash-lite ($0.10/M tokens) — the cheap model

You can add more student models to compare (click "+ Add Student Model").

4 Click "Run Optimization"

This kicks off the full pipeline:

  1. Evaluates the teacher model with all examples (the "monolith" baseline)
  2. Evaluates the student model with no optimization (the "naive" baseline)
  3. Runs the optimizer to find the best prompt for the student
  4. Evaluates the student with the optimized prompt

This takes a few minutes. The progress bar updates as each step completes.

5 Read the results

The results table shows three columns per student model:

  • Monolith: the expensive model doing the task. This is the ceiling — the best possible quality at highest cost.
  • Naive: the cheap model with no optimization. This is the floor.
  • DSPy: the cheap model with the optimized prompt. This should be close to the monolith at a fraction of the cost.

Green highlighted values = best in that row. You want DSPy to match or beat the monolith.

6 Download the production prompt

Click the download button to get a .txt file containing the optimized prompt. This file has:

  • System instructions — use as your system prompt in any API
  • Few-shot examples — include these in the conversation
  • User message template — where you insert each new request

This works with any LLM API (Gemini, OpenAI, Anthropic, etc.). No DSPy dependency needed in production.

Walkthrough: Creating a Fresh Task

When you want to optimize a prompt for your own task:

1 Click "Fresh Task"

This clears everything and gives you a blank form with starter fields.

2 Define your task

Fill in the name, description, and guidelines. Think of the description as "what would you tell a new hire to do?" and the guidelines as "the reference material they'd need."

3 Set up signature fields

Add input fields (what goes in) and output fields (what comes out). Remove the starter fields if they don't match your task. Every output field needs a description that tells the model exactly what format to use.

4 Configure metrics

Add one metric per output field. Pick the type that matches the field:

  • Classification/label → Exact Match
  • List of items → F1 Phrases
  • Free text with rules → Rule Quality

Set weights based on what matters most for your use case.

5 Add training data

Click "+ Add Row" or "Import CSV" to add labeled examples. Each row needs values for ALL fields (inputs and outputs). For rows where an output doesn't apply (e.g., no suggestion needed for compliant copy), use an empty string.

6 Run and iterate

Click "Run Optimization" and review results. If the scores are low, you likely need more examples, clearer guidelines, or better metric configuration.

Common pitfalls:
  • Too few examples (<20) — the optimizer doesn't have enough signal
  • Vague output descriptions — the model guesses the format instead of following it
  • Missing metrics on output fields — the optimizer ignores un-scored fields
  • Inconsistent labeling in examples — "True" vs "true" vs "yes" confuses exact match

Metrics in the results table

ColumnWhat It Means
composite The overall score — a weighted average of all your metrics. This is the single number to compare across configurations.
Individual metrics Each metric you defined shows up as its own row. Useful for seeing WHERE the model is struggling (e.g., good at classification but bad at phrase detection).
latency Average response time in milliseconds. The optimized model may be slightly slower than naive (because the prompt is longer with few-shot examples) but dramatically faster than the monolith.

What "+/-" means

Scores show mean +/- standard deviation across your eval trials. A small "+/-" means the score is stable and reliable. A large "+/-" means the model's output varies between runs — consider using more eval trials or adding temperature=0 to your model config.

What "best" (green) means

The highest score in each row is highlighted green. For latency, the lowest (fastest) value is highlighted. Ideally, the DSPy column should be green for most metrics while also being green for latency.
